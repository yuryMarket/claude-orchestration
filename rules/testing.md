# Правила тестирования

- Всегда писать тесты при добавлении нового функционала или исправлении багов
- Интеграционные тесты приоритетнее моков для БД и внешних сервисов — моки могут маскировать реальные сбои
- Тесты должны быть изолированы: не зависеть от порядка выполнения или разделяемого состояния
- Каждый тест проверяет одно поведение — называть его чётко

## Python (pytest)
- Использовать фикстуры, parametrize, conftest.py
- Именование: `test_<что>_<сценарий>_<ожидание>` (например, `test_create_user_duplicate_email_raises`)
- Целевое покрытие: >80% для нового кода
- Использовать `pytest-cov` для отчётов о покрытии

## TypeScript (vitest / jest)
- Именование: `should <делать что-то> when <условие>`
- Использовать `describe` / `it` для группировки
- Мокать только внешние зависимости (HTTP, файловая система), не внутренние модули
- Предпочитать `vi.fn()` / `jest.fn()` ручным мокам

## Infrastructure as Code
- `terraform validate` + `terraform plan -detailed-exitcode` для каждого изменения
- Использовать `terratest` или `pytest` + subprocess для интеграционного тестирования модулей
- Helm: `helm lint` + `helm template` + `kubeconform` для валидации манифестов

## Quality gates
- Проверка типов + линтинг + тесты должны проходить перед коммитом
- Нестабильные тесты исправлять немедленно, не пропускать
