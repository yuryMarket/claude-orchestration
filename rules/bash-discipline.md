# Дисциплина Bash-вызовов

Разрешённые команды не должны склеиваться с запрещёнными — так permission-система надёжно матчит вызовы, а лишние подтверждения не нужны.

- **Один вызов — один шаг.** Не склеивать самодостаточные команды через `&&` / `;` / `|`: один шаг = один Bash. Исключение — короткие связки, где вторая команда бессмысленна без первой (`mkdir -p X && cp y X/`).
  Плохо: `git status && npm test` → Хорошо: два отдельных вызова Bash.
- **Абсолютные пути вместо `cd dir && cmd`.** Permission-проверка матчит каждый сегмент цепочки; `cd` вне allowlist роняет всю связку в промпт — использовать абсолютные пути.
  Плохо: `cd /repo && pytest tests/` → Хорошо: `pytest /repo/tests/`.
- **Скрипты запускать через Write в файл, не heredoc и не `-c`.** Сначала Write скрипта (проект/скретчпад), затем запуск по абсолютному пути: содержимое видно в диффе, запуск чисто матчится с allow-правилом.
  Плохо: `python3 -c "import os; print(os.listdir())"` → Хорошо: `Write /scratchpad/x.py`, затем `python3 /scratchpad/x.py`.
- **Не смешивать allow и не-allow в одной строке.** Никогда не смешивать команды из allow-списка с командами вне его, включая подстановки `$(…)` и backticks — вся связка требует подтверждения или блокируется.
  Плохо: `kubectl get pods -n $(cat ns.txt)` → Хорошо: прочитать ns отдельным вызовом, затем `kubectl get pods -n <ns>`.
