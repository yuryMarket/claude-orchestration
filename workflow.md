# AIDD Workflow: 8 этапов + Quality Gates

## Этапы

| # | Этап | Роль | Skill | Результат |
|---|------|------|-------|-----------|
| 1 | Идея и PRD | Аналитик | `/idea [ticket] [title]` | `docs/prd/<ticket>.prd.md` |
| 2 | Исследование | Исследователь | `/research [ticket]` | `docs/research/<ticket>.md` |
| 3 | Архитектура | Планировщик | `/plan [ticket]` | `docs/plan/<ticket>.md` |
| 4 | Декомпозиция задач | Task Planner | `/tasks [ticket]` | `docs/tasklist/<ticket>.md` |
| 5 | Реализация | Имплементер | `/implement [ticket]` | Код + тесты + обновлённый таск-лист |
| 6 | Код-ревью | Ревьюер | `/review [ticket]` | Отчёт с blocking/important замечаниями |
| 7 | QA и тестирование | QA-инженер | `/qa [ticket]` | `reports/qa/<ticket>.md` |
| 8 | Документация | Техписатель | `/docs-update [ticket]` | CHANGELOG, summary, обновлённая документация |

## Служебные роли

| Роль | Skill | Назначение |
|------|-------|------------|
| Валидатор | `/validate [ticket]` | Проверка статуса quality gates |
| Оркестратор | `/feature-development [ticket]` | Показать текущее состояние, предложить следующий шаг. НЕ модифицирует код |

## Quality Gates

| Gate | Этап | Условие прохождения |
|------|------|---------------------|
| `AGREEMENTS_ON` | До начала работы | conventions.md, CLAUDE.md, workflow.md, агенты и skills существуют |
| `PRD_READY` | После /idea | PRD заполнен, нет блокирующих вопросов, метрики определены |
| `PLAN_APPROVED` | После /plan | Архитектура описана, NFR/риски задокументированы, ADR согласован |
| `TASKLIST_READY` | После /tasks | Гранулярные задачи с acceptance criteria |
| `IMPLEMENT_STEP_OK` | После /implement | Код + тесты проходят + таск-лист обновлён для каждой задачи |
| `REVIEW_OK` | После /review | CI зелёный, нет blocking-комментариев |
| `RELEASE_READY` | После /qa | QA пройдено, нет критических багов |
| `DOCS_UPDATED` | После /docs-update | Документация соответствует коду, onboarding/runbooks актуальны |

## Правила перехода между этапами

- Нельзя начать этап N+1 пока gate этапа N не пройден
- `/validate` проверяет все gates для текущего тикета
- `/feature-development` показывает статус и рекомендует следующее действие
- `/implement` требует явного подтверждения пользователя перед каждым изменением кода

## Расположение артефактов (уровень проекта, создаются по требованию)

```
docs/
  prd/           # PRD-документы
  research/      # Исследовательские отчёты
  plan/          # Архитектурные планы
  tasklist/      # Списки задач
  .active_ticket # ID текущего тикета
reports/
  qa/            # QA-отчёты
  review/        # Отчёты код-ревью
```

## SRE/DevOps-специфика

Каждый этап включает доменно-специфичные аспекты:

- **Идея/PRD**: Влияние на инфраструктуру, требования к наблюдаемости, влияние на SLA/SLO
- **Архитектура**: План миграции, стратегия отката, план мониторинга, анализ blast radius
- **Реализация**: Валидация IaC (terraform validate/plan), проверка CI/CD пайплайнов, сканирование безопасности
- **QA**: Инфраструктурные тесты, нагрузочные тесты, хаос-инженерия, проверка canary/blue-green
- **Документация**: Runbooks, процедуры реагирования на инциденты, ADR (Architecture Decision Records)
