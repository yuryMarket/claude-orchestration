# Архитектура: config-watcher

**Дата**: 2026-04-06  
**Статус**: реализован полностью

## Проблема

В процессе работы Claude допускает ошибки, пользователь их исправляет. Эти паттерны ошибок нужно фиксировать в конфиг-файлах `~/.claude/` чтобы они не повторялись.

---

## Компоненты

| Компонент | Тип | Файл | Статус |
|---|---|---|---|
| `detect-correction.sh` | bash Stop хук | `~/.claude/hooks/detect-correction.sh` | ✅ |
| `session-start-pending.sh` | bash SessionStart хук | `~/.claude/hooks/session-start-pending.sh` | ✅ |
| `config-watcher` | агент (Sonnet) | `~/.claude/agents/config-watcher/AGENT.md` | ✅ |
| `/config-watch` | скилл | `~/.claude/skills/config-watch/SKILL.md` | ✅ |
| `settings.json` | регистрация хуков | `~/.claude/settings.json` | ✅ |

## Полный поток (автоматический режим)

```
Claude завершает ответ
         ↓
[1] detect-correction.sh (bash, Stop хук)
    Проверяет ДВА условия одновременно:
    • последнее сообщение пользователя содержит слово-коррекцию
    • И в транскрипте ЛЮБОЙ assistant-ответ содержал Edit/Write
      (не только последний — проверяется весь transcript_path)
    Одно из двух → exit 0, ничего не делаем
         ↓ оба условия выполнены
[2] Проверяем /tmp/claude-watcher-ran-{session_id}
    Файл существует → rm флага → exit 0 (защита от анализа ответов watcher-а)
         ↓ файла нет
    Записываем transcript_path в /tmp/claude-correction-flag-{session_id}
    Инициализируем /tmp/claude-cursor-{session_id} если не существует
    (cursor = текущее кол-во сообщений, следующий анализ начнётся с этой позиции)
    Возвращаем {"ok": false} — Claude продолжит и запустит config-watcher
         ↓
[3] config-watcher агент — читает дельту транскрипта (cursor → конец файла)
         ↓
[4] Быстрый фильтр (эвристика, без глубокого анализа)
    "Системная проблема или разовая?"
    Разовая → обновить cursor, {"ok": true}, выход
         ↓ системная
[5] Детальный анализ
    Что было неправильно? В каком файле фиксировать?
    Определить уровень: глобальный (~/.claude/) или проектный (./.claude/)
         ↓
[6] Проверка целевого файла

    Похожего правила нет → к шагу [7]

    Похожее правило есть, но проблема возникла всё равно
         → анализируем почему не сработало:
            • слишком абстрактно — нет конкретики
            • нет примера — непонятно как применять
            • противоречит другому правилу
            • правило не в том файле
         → предлагаем переписать (diff: было → стало) → к шагу [7]

    Похожее правило есть и решает другую грань
         → предлагаем дополнить → к шагу [7]
         ↓
[7] Проверяем /tmp/claude-proposed-{session_id}
    Уже предлагали похожее? → обновить cursor, {"ok": true}, выход
         ↓ не предлагали
[8] Показываем предложение пользователю
    Три формата: новое правило / переписать / дополнить
    Записываем в ~/.claude/pending-changes.md
    Записываем в /tmp/claude-proposed-{session_id}
    touch /tmp/claude-watcher-ran-{session_id}
    Возвращаем {"ok": false, "reason": "..."}
         ↓
[9] Пользователь отвечает
    "да" → вносим изменение → удаляем из pending-changes.md → обновляем cursor
    "нет" → удаляем из pending-changes.md → обновляем cursor
    Если pending-changes.md пустой → удалить файл
```

---

## Ручной вызов: /config-watch

Шаги [1] и [2] пропускаются — решение о запуске уже принято пользователем.  
Агент запускается в Режиме B и начинает **с шага [3]**.

```
/config-watch вызван
         ↓
Если ~/.claude/pending-changes.md существует и не пустой:
    Показать содержимое → предложить обработать каждую запись
    (применить / отклонить / пропустить)
         ↓
[3] Читает ВЕСЬ транскрипт текущей сессии (cursor = 0, с начала)
[4–8] Быстрый фильтр → детальный анализ → предложение
[9] Обработать ответ пользователя
```

Если системных проблем не найдено — сообщить: *"Анализ завершён, предложений нет."*

## Файлы на сессию (в /tmp, удаляются при перезагрузке)

| Файл | Содержит | Создаётся в |
|---|---|---|
| `claude-cursor-{session_id}` | номер последнего проанализированного сообщения | [2] detect-correction.sh (при первом trigger) |
| `claude-proposed-{session_id}` | краткие описания уже предложенных изменений | [8] config-watcher агент |
| `claude-watcher-ran-{session_id}` | флаг что агент уже отработал после этого Stop | [8] config-watcher агент |
| `claude-correction-flag-{session_id}` | путь к transcript_path | [2] detect-correction.sh |

## Постоянные файлы (между сессиями)

| Файл | Содержит |
|---|---|
| `~/.claude/pending-changes.md` | предложения, не подтверждённые/отклонённые до закрытия сессии |

---

## Очистка pending-changes.md

| Когда | Действие |
|---|---|
| Пользователь ответил "да" | Запись удаляется, изменение вносится |
| Пользователь ответил "нет" | Запись удаляется |
| SessionStart → "применить" | Изменения вносятся, запись удаляется |
| SessionStart → "отклонить" | Запись удаляется |
| SessionStart → "позже" / продолжи работу | Запись остаётся |
| Файл стал пустым | Файл удаляется полностью |

## Защиты от проблем

| Проблема | Защита |
|---|---|
| Ложные срабатывания | Двойное условие: слово-коррекция + Edit/Write в транскрипте |
| Разовая ошибка vs системная | Быстрый фильтр (шаг [4]) |
| Повтор анализа одних сообщений | cursor в /tmp |
| Дублирование предложений в сессии | /tmp/claude-proposed-{session_id} |
| Правило есть но не работает | Анализ причины + предложение переписать |
| Дублирование в целевом файле | Чтение файла + семантическое сравнение |
| Анализ ответов самого watcher-а | /tmp/claude-watcher-ran-{session_id} |
| Потеря предложений при закрытии | pending-changes.md |
| Зависание хука | timeout: 60 в settings.json |

## Известные ограничения реализации

### 1. Condition B проверяет весь транскрипт, не только последний ответ

`detect-correction.sh` ищет `Edit`/`Write` во **всех** assistant-сообщениях транскрипта:
```bash
jq -r 'select(.message.role == "assistant") | .message.content[]? | select(.type == "tool_use") | .name' "$TRANSCRIPT_PATH"
```
Это значит: если Claude редактировал файл 10 сообщений назад, а сейчас пользователь написал "нет, это неправильно" по другому поводу — хук всё равно сработает. Агент в шаге [4] должен это отфильтровать как "разовую" проблему.

### 2. Обработка pending-changes.md при SessionStart — только нотификация

`session-start-pending.sh` только **показывает** содержимое `pending-changes.md` и печатает инструкции. Фактическое применение/отклонение требует либо:
- Вызова `/config-watch` (агент обрабатывает pending в начале Режима B)
- Явного запроса к Claude ("применить pending changes") — обрабатывается main-инстансом Claude без специального агента

Автоматической обработки pending при SessionStart **нет**.

### 3. Переменные окружения в SKILL.md

`/config-watch` передаёт агенту `$CLAUDE_SESSION_ID` и `$CLAUDE_TRANSCRIPT_PATH` как литеральные строки в промпте. Эти переменные корректно раскрываются только если скилл вызывается в контексте где эти env-переменные доступны. В текущей реализации агент работает с общим general-purpose типом и получает эти значения через подстановку в тексте промпта.

### 4. Режим A не передаёт данные агенту через stdin

В архитектурном плане предполагалось что `config-watcher` агент получает данные через stdin (как Stop хук). Фактически:
- `detect-correction.sh` возвращает `{"ok": false}` — Claude продолжает
- Claude сам вызывает config-watcher агент через `Agent` tool
- Агент читает /tmp файлы напрямую через `Read` tool (не stdin)

Это рабочая схема, но она означает что агент зависит от корректно выставленных /tmp файлов, а не от прямой передачи данных.
