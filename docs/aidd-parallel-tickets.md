# Параллельная работа над несколькими тикетами AIDD

**Дата**: 2026-06-01
**Обновлено**: 2026-06-02 (AIDD-002)
**Тикет**: AIDD-001, AIDD-002

---

## Суть

С версии AIDD-001 каждая сессия Claude Code хранит **свой** активный тикет независимо от других
сессий, открытых в той же директории. Можно одновременно вести FEAT-042 в одном окне терминала
и BUG-017 в другом — они не мешают друг другу.

---

## Как открыть вторую сессию на другой тикет

1. Открой новый терминал (или новую вкладку iTerm/tmux).
2. Перейди в ту же рабочую директорию: `cd ~/core-ai` (или любой другой проект).
3. Запусти `claude` (или `claude --resume` для продолжения прерванной сессии).
4. Выполни `/idea BUG-017 Исправить X` — скилл запишет тикет `BUG-017` именно в эту сессию.

Первая сессия продолжает работать с FEAT-042, вторая — с BUG-017. `gate-workflow.sh` в каждой
сессии блокирует только те правки, которые нарушают gates активного тикета этой конкретной сессии.

---

## Почему сессии изолированы

Хук `session-start.sh` при старте:
- читает `session_id` из stdin JSON (присутствует во всех версиях Claude Code CLI);
- создаёт файл `~/.claude/sessions/aidd/<session_id>.json` с полями
  `{ticket, cwd, source, created, updated}`.

Файл привязан к конкретному `session_id` — уникальному идентификатору сессии, который Claude Code
выдаёт при запуске и сохраняет при resume/compact. Две параллельные сессии получают разные
`session_id`, поэтому пишут в разные файлы — гонка за общим ресурсом устранена.

Когда `gate-workflow.sh` проверяет gates, он сначала смотрит в файл своей сессии
(`source=session:<ID>`), а если не находит — падает в legacy `docs/.active_ticket`.

---

## Цепочка резолюции тикета (двухуровневая)

С AIDD-002 цепочка стала двухуровневой:

```
1. PRIMARY:  ~/.claude/sessions/aidd/<session_id>.json
             └── guard: .cwd == cwd текущей сессии  → source=session
2. FALLBACK: <cwd>/docs/.active_ticket  (legacy plain-text) → source=legacy
3. NONE:     ничего → AIDD неактивен (exit 0, не блокируем)
```

Бывший уровень 3 (`<cwd>/.claude/docs/.active_ticket`) **удалён** — он никем не писался и
мог возвращать устаревший тикет. Единственный canonical legacy-путь теперь —
`<cwd>/docs/.active_ticket`.

---

## Условная запись legacy через `aidd-write-legacy.sh`

Запись в `docs/.active_ticket` выполняется через helper:

```bash
bash ~/.claude/hooks/_lib/aidd-write-legacy.sh "<cwd>" "<ticket>"
```

Helper пишет файл **только если**:

- файл пуст или не существует (первая запись), **или**
- файл уже содержит тот же тикет (идемпотентная перезапись).

Если файл содержит **другой** тикет (признак параллельной сессии, которая уже записала свой
тикет туда) — запись пропускается, в stderr выводится предупреждение:

```text
AIDD: legacy docs/.active_ticket=<X> принадлежит другой сессии — пропускаю запись <ticket>
```

Это означает, что при двух параллельных сессиях в одной директории `docs/.active_ticket`
хранит тикет той сессии, которая первой его записала. Вторая сессия не перетирает его. Обе
сессии при этом изолированы через per-session файл (уровень 1) — legacy-файл служит fallback
только для новых и унаследованных сессий.

Helper всегда завершается с exit 0 (fail-open).

---

## Что происходит при `/clear`

Команда `/clear` очищает контекст и создаёт **новый** `session_id`. При этом:

- `session-start.sh` срабатывает с `source=clear`;
- видит, что per-session файла для нового id нет;
- читает тикет из legacy `docs/.active_ticket` (если он записан) и **наследует** его в новый
  per-session файл.

Результат: сессия продолжает работать с тем же тикетом, что был до `/clear`. Если в
`docs/.active_ticket` ничего нет (или там тикет другой сессии) — унаследуется то, что там есть.
Это сознательный компромисс: `/clear` — редкая операция.

Если унаследованный тикет не тот — просто выполни `/idea <правильный-тикет>` в новой сессии.

---

## Legacy fallback для старых CLI

Если `session_id` недоступен (CLI < v2.1.132 без переменной `$CLAUDE_CODE_SESSION_ID`, или
отсутствует `python3`), AIDD автоматически деградирует к единому `docs/.active_ticket`. Это
означает, что параллельные сессии снова будут делить один указатель — как раньше, до AIDD-001.
Функциональность не сломана, изоляция просто не работает.

---

## Диагностика

В статус-панели при старте сессии теперь видно:

```text
AIDD session_id: a1b2c3d4-...
Active ticket: FEAT-042
Source: session (или legacy)
```

Если `Source: legacy` — сессия использует старый путь. Если `Source: session` — изоляция активна.

В `gate-workflow.sh` при каждой операции Edit/Write пишется в stderr:

```text
AIDD: ticket=FEAT-042 source=session:a1b2c3d4-...
```

или:

```text
AIDD: ticket=FEAT-042 source=legacy
```

Источник `legacy` теперь однозначен: только `docs/.active_ticket` (второго legacy-пути нет).

---

## Очистка старых файлов сессий

Файлы в `~/.claude/sessions/aidd/` очищаются автоматически при каждом старте новой сессии
(TTL = 14 дней по mtime). Вручную очищать не нужно. Директория и её содержимое — приватны
(только ticket ID и путь к директории проекта, никаких секретов).

---

## Migration note (AIDD-002)

Путь `<проект>/.claude/docs/.active_ticket` (бывший уровень 3) **больше не читается** ни одним
из хуков. Если вы вручную создавали тикет-указатель по этому пути — перенесите его в
`<проект>/docs/.active_ticket`.
